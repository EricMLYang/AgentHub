# Weekly Updates 週更機制

> 保持團隊透明與同步的最低成本方式

---

## 為什麼要週更？

- **透明**：讓大家知道每個人在做什麼、卡在哪裡
- **問責**：確保每個 Agent 想法都有人持續推進
- **求助**：早點發現卡點，早點尋求協助
- **低摩擦**：每週 15 分鐘，3 行也可以

---

## 週更規則

### 1️⃣ 每週至少更新一次
- 不用寫長篇報告，3 行也可以
- 用 [Weekly Update Issue Template](../../issues/new/choose)
- 標題格式：`[Weekly] 2026-W02 - 你的名字`

### 2️⃣ 必填內容（超短）
- **本週進度**：做了什麼（1~3 點）
- **卡點**：遇到什麼困難（1 點，沒有就寫「無」）
- **下週步驟**：下週要做什麼（1 點）
- **需要協助**：需要誰/什麼資源（沒有就寫「無」）

### 3️⃣ 沒進度也要更新
- 如果這週沒進度，就寫「無進度，原因是 [...]」
- 透明比好看更重要

---

## 如何寫週更

### ✅ 好的週更範例

**本週進度**：
- 完成 PR 審查 Agent 的 prototype
- 測試了 5 個 PR，確認基本流程可行
- 寫了初步的驗收標準

**卡點**：
無法成功呼叫 GitHub API，一直遇到 401 錯誤

**下週步驟**：
解決 API 認證問題，成功在 GitHub Action 中呼叫 API

**需要協助**：
需要有 repo admin 權限的人幫忙檢查 token 設定

---

### ❌ 不好的週更範例

**本週進度**：
- 研究了一些東西
- 做了一些測試
- 進行中

**卡點**：
有點困難

**下週步驟**：
繼續做

**需要協助**：
無

**問題在哪**：
- 「研究了一些東西」→ 太模糊，研究了什麼？
- 「做了一些測試」→ 測試了什麼？結果如何？
- 「有點困難」→ 困難在哪？
- 「繼續做」→ 做什麼？

## 週更節奏

### 選項 1：Async 模式（建議）
- 每人在**週五下班前**更新
- 用 Issue comment 或新的 Weekly Issue
- 其他人有空時回覆或提供協助

### 選項 2：同步週會
- 每週固定時間（例如週五下午）
- 每人 3 分鐘快速分享
- 會議紀錄貼到 weekly/ 資料夾

---

## 如何使用 Copilot 輔助

如果你的筆記很散亂，可以用 Copilot 幫你整理：

1. 到 [write_weekly_update.md](../.github/copilot/prompts/write_weekly_update.md)
2. 複製 prompt
3. 貼到 Copilot Chat
4. 附上你的散亂筆記
5. 讓 Copilot 產出結構化的週更
6. 檢查並調整後貼到 Issue

## 週更範本檔案

每週的週更可以整理到 `weekly/` 資料夾，格式：

- `weekly/2026-W02.md` — 第 2 週的週更摘要
- `weekly/2026-W03.md` — 第 3 週的週更摘要

也可以只用 Issue + Label `weekly`，不需要額外建檔案。

## FAQ

**Q: 我這週真的沒進度怎麼辦？**  
A: 就誠實寫「無進度」+ 原因。可能是：
- 被其他專案拉走
- 這個想法需要重新評估
- 等待某個資源或審批

**Q: 我可以不寫卡點嗎？**  
A: 如果真的沒有，就寫「無」。但如果進度慢，通常都有卡點，寫出來可以得到幫助。

**Q: 下週步驟可以寫「繼續做」嗎？**  
A: 最好不要。改成「完成 X」或「測試 Y」，越具體越好。

**Q: 我可以跳過一週嗎？**  
A: 可以，但如果連續 2 週沒更新，這個 Idea 可能就被視為「暫停」或「取消」。
