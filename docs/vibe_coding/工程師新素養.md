# AI 時代工程師必備能力地圖（2026）

> 核心前提：AI 接手了「產碼速度」，人類的價值轉向 **判斷力、系統思維、驗證能力**。
> 以下每個角色只列 **3 項最高槓桿的能力**，是 AI 最難替代、且缺乏時最容易出事的。

---

## 所有角色共通（先讀這段）

不論角色，2026 年每位工程師都需要內建一個底層習慣：

> **「AI 產出的東西，你敢不敢用你的名字簽核上線？」**
>
> 如果你無法解釋這段程式碼為什麼是對的，它就不該被合併。
> 這不是態度問題，是 GitClear 211M 行程式碼追蹤出來的結論：
> 團隊 review 深度已下降 30%，copy/paste 首次超越 refactoring，技術債正在無聲累積。

## 前端工程師

| # | 能力 | 為什麼是你 | 具體行動 |
|---|------|-----------|---------|
| 1 | **辨識「外觀正確」的陷阱** | AI 生成的 UI 程式碼格式完美、命名規範，但隱藏 XSS、不安全的 DOM 操作、外洩的 state。前端是使用者直接觸及的攻擊面 | 每次 review AI 生成的元件，手動檢查：輸入是否有 sanitize？敏感資料是否暴露在 client state？第三方 CDN/script 是否可信？ |
| 2 | **依賴審計直覺** | 前端生態依賴樹極深（一個元件拉進數十個 transitive dependencies），AI 最容易在這裡推薦幻覺套件或過時版本 | 對每個 AI 建議的新 npm package：查 npmjs.com 確認存在、看 weekly downloads、檢查最後更新日期、比對是否與知名套件撞名 |
| 3 | **無障礙與效能的系統判斷** | AI 能生成「能跑」的 UI 但幾乎不會主動考慮 a11y、bundle size、render performance。這些是 AI 的系統性盲點 | 建立 Lighthouse / Core Web Vitals 基線，AI 生成的頁面必須通過效能門檻；a11y 檢查（axe / screen reader 測試）列入 CI |

## 後端工程師

| # | 能力 | 為什麼是你 | 具體行動 |
|---|------|-----------|---------|
| 1 | **故障場景想像力** | AI 寫的 happy path 完美，但不會主動思考：如果下游 timeout？如果 retry 風暴？如果 partial failure？這是 AI 最致命的盲點 — 它不會主動破壞自己的程式碼來測試韌性 | 每個 API/服務變更，強迫自己回答：「如果這個依賴掛了，會發生什麼？」並確認 circuit breaker/fallback/timeout 沒有被 AI 誤刪或改壞 |
| 2 | **授權與安全邊界意識** | 跨服務的 JWT/header/claims 變更是 AI CR 最難抓的問題（靜默授權回退）。45% AI 程式碼含安全漏洞，後端是主戰場 | 任何涉及 auth 的變更，不信任 AI CR 結論，親自寫 contract test 確認跨服務的 token 傳遞與權限檢查沒有靜默降級 |
| 3 | **資料操作的不可逆意識** | DELETE/TRUNCATE/MERGE 在 AI 看來只是一行程式碼，但在生產環境可能是災難（Replit 事件的教訓） | 養成肌肉記憶：看到任何資料寫入/刪除操作 → 自問「有 dry-run 嗎？有備份嗎？能 rollback 嗎？」沒有就不過 |

## 架構師 / Tech Lead

| # | 能力 | 為什麼是你 | 具體行動 |
|---|------|-----------|---------|
| 1 | **跨服務影響的全局視野** | AI 只看到單一 PR 的上下文，看不到改動在整個系統中的漣漪效應。架構師是唯一有全局 mental model 的人 | 對每個 R2/R3 PR，畫出（或腦中走過）影響的服務鏈，特別檢查：schema 變更的向後相容、event contract 的消費者、shared state 的一致性 |
| 2 | **技術債的主動治理** | AI 加速了 feature delivery 但同時加速了 architectural erosion。如果架構師不主動排程 refactoring，沒有人會做 | 每個迭代保留 15-20% 容量給 refactoring；定期 review duplication 指標；拒絕「AI 生成所以先這樣」的技術債藉口 |
| 3 | **AI 工具鏈的風險治理** | 團隊用什麼 AI tool、tool 有什麼權限、連接了什麼外部服務 — 這是架構師的責任範圍，不是個人選擇 | 維護團隊的 AI tool allowlist、定義 agent 權限邊界、確保 MCP server / IDE extension 納入安全審計範圍 |

## 數據工程師 / 分析師

| # | 能力 | 為什麼是你 | 具體行動 |
|---|------|-----------|---------|
| 1 | **效能直覺與成本意識** | AI 寫的 Spark/SQL 在小樣本上完美運行，但生產環境可能觸發 data skew、shuffle explosion、全表掃描，帳單暴增。AI 沒有成本概念 | 任何 AI 生成的 query/job，必須附 explain plan 或 Spark UI 截圖。養成習慣問：「這個 join 的基數是多少？partition 策略對嗎？」 |
| 2 | **資料品質的懷疑本能** | AI 生成的 pipeline 可能產出「格式正確但語義錯誤」的資料。分析師如果不質疑數字的合理性，下游決策就會建立在錯誤基礎上 | 對 AI 生成的 ETL/transform 邏輯，用已知答案的樣本做 sanity check；對聚合結果建立 data quality assertion（值域、null 率、row count 預期） |
| 3 | **Incremental processing 的邊界條件思維** | AI 寫 watermark/CDC/merge 邏輯時，最容易忽略的是邊界情況：late-arriving data、exactly-once 語義、重播冪等性 | 每次改動 incremental 邏輯，必須附 replay test 結果。問自己：「如果同一批資料跑兩次，結果一樣嗎？如果資料遲到 3 小時呢？」 |

## DevOps / SRE

| # | 能力 | 為什麼是你 | 具體行動 |
|---|------|-----------|---------|
| 1 | **CI/CD pipeline 即攻擊面的意識** | AWS CodeBuild webhook 漏洞（CodeBreach）證明 pipeline 配置本身就是供應鏈攻擊的入口。AI agent 的 token scope 是你的守備範圍 | 定期審計 pipeline 中的 token scope、webhook filter、build trigger 條件；AI agent 的 credential 一律最小權限 + 定期輪換 |
| 2 | **可觀測性設計** | AI 產碼時幾乎不會主動加入 structured logging、metrics、alerting。如果 SRE 不把可觀測性當 gate，上線後出事就是盲飛 | 將「有 metrics/alert 更新」列入 R2/R3 的合併必要條件；確保每個新服務/endpoint 有 baseline SLO 和對應告警 |
| 3 | **Rollback 的肌肉記憶** | AI 不會幫你設計 rollback strategy。當 AI agent 的多步驟部署出問題時，能在幾分鐘內回滾是 SRE 獨有的價值 | 每個部署變更必須附具體的回滾命令/步驟；定期演練 rollback（不只是寫在文件上）；agentic deployment 必須支援原子性回滾 |

## QA / 測試工程師

| # | 能力 | 為什麼是你 | 具體行動 |
|---|------|-----------|---------|
| 1 | **「AI 沒測到什麼」的逆向思維** | AI 生成的測試覆蓋率數字好看，但幾乎只測 happy path。真正的品質守門人是能想到 AI 想不到的 edge case 的人 | 對 AI 生成的 test suite，系統性補充：邊界值、null/empty、併發、故障注入、跨時區、權限不足場景 |
| 2 | **Contract testing 實作能力** | 跨服務的靜默回退是 AI 時代最隱蔽的 bug 類型。QA 如果只做 UI E2E，抓不到 header/schema 不一致 | 學會並推動 Pact 或類似的 consumer-driven contract testing 框架，確保每次 schema 變更都有跨服務驗證 |
| 3 | **對「AI 說通過」的健康懷疑** | AI CR 和 AI test generation 都有確認偏誤 — 傾向同意自己生成的程式碼。QA 是打破這個迴圈的角色 | 建立獨立於 AI 的驗證流程：至少一輪測試由人類從需求出發設計（而非從 AI 程式碼出發），避免測試和程式碼犯相同的錯 |

## 一頁速查：各角色的一句話提醒

| 角色 | 你最不可被 AI 取代的價值 |
|------|------------------------|
| 前端 | 你能看穿「畫面正常」背後的安全與效能問題 |
| 後端 | 你能想像「如果這個掛了」的故障場景 |
| 架構師 | 你能看到單一 PR 看不到的系統漣漪 |
| 數據工程 | 你能質疑「數字對嗎」而不是只看「能跑嗎」 |
| DevOps/SRE | 你能在出事的前 5 分鐘把系統救回來 |
| QA | 你能想到 AI 想不到的壞掉方式 |
