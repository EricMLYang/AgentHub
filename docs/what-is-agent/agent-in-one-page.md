# 什麼是 Agent？（一頁搞懂）

## 一句話定義

> **Agent 是一個可以「自主決策、調用工具、持續調整行為」來完成目標的 AI 系統。**

### 可用的 Agent 特徵
- 能根據環境變化調整行為
- 可以自主選擇和組合多個工具
- 有明確的目標，但達成路徑不固定
- 能處理模糊、動態的需求

### 不是 Agent 的情況
- 固定步驟的自動化流程（那是 Workflow）
- 單純回答問題的 LLM（那是 Chatbot）
- 只執行預定命令的腳本（那是 Script）

---

## Agent vs Workflow vs Script vs RAG

| 類型 | 決策方式 | 工具使用 | 適用場景 | 範例 |
|------|---------|---------|---------|------|
| **Script** | 無（固定邏輯） | 預定的命令序列 | 完全確定的重複任務 | 每天備份資料庫 |
| **Workflow** | 有限分支 | 預先配置的工具鏈 | 已知的業務流程 | 請假審批流程 |
| **RAG** | 檢索 + 回答 | 查詢向量資料庫 | 需要知識檢索的 Q&A | 文件問答機器人 |
| **Agent** | 動態規劃 | 自主選擇工具組合 | 複雜、不確定、需探索 | 自動化程式碼審查與修正 |

---

## 我們團隊的 Agent 例子

### 例子 1：Pull Request 自動審查 Agent
**痛點**：每次 PR 都要手動檢查格式、測試覆蓋率、安全漏洞  
**Agent 做什麼**：
- 分析 PR diff
- 自動執行 linter、測試、安全掃描
- 根據結果決定要：自動合併 / 留評論 / 標記需人工審查
- 如果測試失敗，會建議修復方法

**為什麼是 Agent**：因為它要「判斷嚴重性」、「決定下一步」、「組合多個工具」

---

### 例子 2：文件更新 Agent
**痛點**：API 改了但文件總是忘記更新  
**Agent 做什麼**：
- 偵測到 API 變更（git diff）
- 找出受影響的文件（AST 分析 + 向量搜尋）
- 生成更新建議
- 如果有信心就直接發 PR，沒信心就標記 `needs-review`

**為什麼是 Agent**：因為它要「搜尋關聯」、「評估信心度」、「自主決定行動」

---

### 例子 3：客戶問題分類與轉派 Agent
**痛點**：客服訊息太多，不知道要轉給誰  
**Agent 做什麼**：
- 讀取客戶訊息
- 判斷問題類型（技術 / 帳務 / 功能需求）
- 查詢相關知識庫（RAG）
- 決定：自動回覆 / 轉人工 / 建立 Issue

**為什麼是 Agent**：因為它要「分類意圖」、「組合檢索和規則」、「動態決策」

---

## 關鍵提醒

✅ **Agent 適合**：複雜、動態、需要探索的任務  
❌ **Agent 不適合**：流程固定、可完全預測的任務（用 Workflow 或 Script 更可靠）
