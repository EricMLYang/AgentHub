# 什麼時候不要用 Agent

## 核心原則

> **如果流程可以明確定義、完全預測、輕鬆測試，那就不要用 Agent。**

Agent 增加了「不確定性」，如果你的任務本來就很確定，用 Agent 只會讓事情變複雜。

---

## 不該用 Agent 的情境

### ❌ 情境 1：規則明確、步驟固定
**例子**：  
- 每天凌晨備份資料庫 → 用 **cron job**
- 收到特定 email 就轉發給某人 → 用 **email filter**
- PR merge 後自動部署 → 用 **CI/CD pipeline**

**為什麼不用 Agent**：這些任務沒有「決策空間」，用固定流程更快、更穩定、更好測試。

---

### ❌ 情境 2：需要 100% 可靠性與可審計性
**例子**：  
- 金融交易的資料驗證
- 醫療數據的合規檢查
- 法律文件的審批流程

**為什麼不用 Agent**：Agent 有「不確定性」，如果失敗會造成法律 / 金錢 / 安全風險，就該用確定性的規則引擎或人工審核。

---

### ❌ 情境 3：只是為了炫技
**例子**：  
「我們來做一個 Agent 來幫忙開關燈吧！」  
→ 其實只是 `if 時間 == 晚上 then 開燈`，根本不需要 LLM

**為什麼不用 Agent**：如果用一個簡單的 if-else 或現成工具就能搞定，用 Agent 只是浪費資源和時間。

---

### ❌ 情境 4：沒有清楚的 Input / Output / Acceptance
**例子**：  
「我想做一個 Agent 來優化團隊生產力」  
→ 什麼叫優化？怎麼衡量？做到什麼算成功？

**為什麼不用 Agent**：如果連驗收標準都說不清楚，Agent 不會比你更清楚。先把需求釐清，再決定要不要用 Agent。

---

## 常見失敗模式

### 😵 失敗 1：為了 AI 而 AI
**症狀**：「這個用 AI 聽起來很酷」  
**結果**：花了 2 週做 Agent，效果還不如一個 10 行的腳本

---

### 😵 失敗 2：過度設計
**症狀**：「我們要讓 Agent 能處理所有可能情況」  
**結果**：Agent 太複雜，無法測試，也無法維護

---

### 😵 失敗 3：沒有降級方案
**症狀**：「Agent 會自己處理，不用擔心」  
**結果**：Agent 出錯時沒有 fallback，導致整個流程卡住

---

## 什麼時候該用 Agent？

回到正面清單：

✅ **任務複雜且動態**（例如：程式碼審查、文件更新、客服分流）  
✅ **有明確的 Input / Output / Acceptance**（可驗證）  
✅ **失敗成本可控**（有 fallback 或人工介入機制）  
✅ **真的能省時間或提升品質**（不是為了炫技）
